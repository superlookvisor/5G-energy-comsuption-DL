# Fallback audit（prevday / roll24 / current）

本报告统计日前特征构造中 `prevday_samehour -> roll24 -> current -> zero` 回退链的实际触发比例。

## 关键读法

- `*_ratio_prevday/roll24/current/zero`：某个变量在该 horizon 上，最终主要落到哪一档（按链优先级）。
- `fallback_prevday_to_roll24_*_ratio`：prevday 缺失但 roll24 存在的比例。
- `fallback_roll24_to_current_*_ratio`：prevday 和 roll24 都缺失但 current 存在的比例。

## 一句话结论（先看这个）

- **prevday_samehour 的可用性并不高**：对 `load_mean/load_pmax`，`prevday` 只覆盖约 **57%–73%**（随 horizon 变化），其余大多需要回退到 `roll24`（约 **24%–40%**），还有一小部分只能用 `current`（约 **2.65%**）。
- `**current` 的占比几乎对所有 horizon 都固定为 2.65%**：这提示存在一批“prevday 与 roll24 都缺失、但 current 不缺失”的固定样本群（通常与某些 BS 的时间断档/窗口不足有关）。
- `**load_std` 更依赖 roll24**：`load_std` 的 `roll24` 占比更高，且 `current` 几乎不会成为最终来源（本次为 0）。

## 指标口径（避免误解）

- **统计对象**：每个 `horizon` 下，对所有 `origin_time`（一般是每天 00:00 起报点）×所有 BS 的样本，统计在构造代理变量时“最终使用了哪一档来源”。
- **对齐方式**：`prevday_samehour` 按时间戳对齐到 `(BS, target_time - 24h)`；不是简单的行序号 `-24`。
- **roll24 的含义**：`roll24` 指 `origin_time` 时刻上已预先计算好的 24 小时滚动统计（例如 `load_mean_roll24`），它不是 “target_time 的 roll24”。

### 表格说明（主要来源占比）

- **horizon**：预测提前量（小时），`target_time = origin_time + horizon`。
- **n**：该 horizon 的样本数（本次每个 horizon 都是 3325，说明每个 horizon 的可用起报点数量一致）。
- `***_ratio_prevday/roll24/current/zero` 的严格定义**（以 `load_mean` 为例）：
  - **prevday**：`load_mean_prevday_samehour` 不缺失，最终直接使用 prevday。
  - **roll24**：prevday 缺失且 `load_mean_roll24` 不缺失，最终使用 roll24。
  - **current**：prevday 与 roll24 都缺失，但 `load_mean`（current）不缺失，最终使用 current。
  - **zero**：前三者都缺失，最终置 0（本次为 0）。
- **校验关系**：同一变量在同一 horizon 下应满足  
ratio{prevday} + ratio{roll24} + ratio{current} + ratio{zero} \approx 1。

### 怎么解读你这份结果（结合数值讲清楚）

- **prevday 覆盖随 horizon 变大整体下降**：`load_mean_ratio_prevday` 从 `h=1` 的 **0.7326** 下降到 `h=24` 的 **0.5732**；对应 `roll24` 从 **0.2409** 上升到 **0.4003**。
  - 直觉解释：`prevday_samehour` 对齐到 `target_time-24h = origin_time + horizon - 24h`。当 horizon 越大，这个对齐点越“靠近 origin_time 之后的区域”，更容易落在数据断档/窗口不足的时段，因此 prevday 更容易缺失。
- **current 恒定 2.65%**：`load_mean_ratio_current` / `load_pmax_ratio_current` 对所有 horizon 都是 **0.0265**。
  - 含义：存在一批样本在所有 horizon 下都同时缺失 prevday 与 roll24（但 current 有），常见原因是某些 BS 在数据起始段/长缺口后刚恢复，导致 24h 窗口未形成。
- `**load_std` 不走 current**：`load_std_ratio_current=0`，说明 std 这条链在 roll24 阶段基本都能兜住（或 current 的 std 在该口径下不作为有效兜底）。

### 你这份结果的结论要点（更直观）

- **prevday→roll24 的回退比例很高**：对 `load_mean/load_pmax`，`h=1` 已有 **0.2409**，到 `h=24` 上升到 **0.4003**。这意味着“用到 prevday_samehour 的样本不足 2/3”，模型在不少情况下依赖 roll24 这种更粗的历史代理。
- **roll24→current 恒定为 0.0265**：说明存在稳定的“prevday 与 roll24 同时缺失但 current 存在”的样本群。建议你后续重点把这批样本定位出来（按 BS/日期），因为它往往对应数据断档或窗口不足的站点。
- `**load_std` 的 roll24 兜底更重要**：`fallback_prevday_to_roll24_load_std_ratio` 从 **0.2674** 到 **0.4268**，而 `fallback_roll24_to_current_load_std_ratio` 恒为 0，说明 std 的缺失基本都能在 roll24 处被修复。

## 对 `run_phaseB_dayahead_only.py` 的影响（是否需要改代码）

- **不需要改主流程的回退链写法**：它本来就是为“prevday 经常缺失”这种现实情况设计的（缺失就回退），所以你的结果不会导致代码逻辑不成立。
- **但需要把它当成“模型难度/不确定性”提示**：
  - 当 `roll24` 占比接近 40% 时，`*_hat` 的“同小时信息”会变弱，峰谷跟随能力可能下降。
  - `current` 虽然只占 2.65%，但如果集中在少量 BS/少量日期，可能形成局部系统性偏差（比如某些站在某几天整体预测偏差大）。

## 建议你下一步怎么用这份审计

- **定位 `current` 样本**：用 `fallback_per_row.csv` 筛 `src_load_mean == "current"`（或 `src_load_pmax == "current"`），看这些样本集中在哪些 BS/日期；再决定是否在 PhaseB 训练时用 `--exclude-bs-csv` 剔除。
- **必要时提高 BS 过滤强度**：例如提高 `--min-merged-obs-per-bs`，能提升 `roll24` 的有效性与 prevday 覆盖（代价是样本/站点变少）。
